# Roundtable — 専門家円卓会議

**重要な判断をするとき、自分ひとりの視点では不安になったことはありませんか？**

コードの設計方針、事業戦略、デザインの方向性——答えが1つに定まらない問いほど、多角的な視点が必要です。でも、専門家を何人も集めて議論するのは現実的ではない。

Roundtable はこの問題を解決します。

あなたの議題に応じて、世界最高レベルの専門家を自動で選定し、構造化された議論を行い、多角的な評価・提言をまとめます。ただ意見を集めるのではなく、対立点を深掘りし、少数意見を見逃さない仕組みが組み込まれているので、「みんな同じことを言って終わる」ということがありません。

---

## こんな方に

- **設計・技術判断で迷う方** — アーキテクチャの選択肢が複数あり、どれが正解かわからない
- **事業やプロダクトの方向性を決めたい方** — 1人で考えていると偏りが出る気がする
- **レビューやフィードバックが欲しい方** — Webサイト、記事、デザイン、なんでも多角的に評価したい
- **見落としやリスクを洗い出したい方** — 自分では気づけない盲点が怖い

専門知識は必要ありません。議題を伝えるだけで、専門家の選定から議論の進行まですべて自動で行います。

## はじめかた

### Claude Code

Claude Code の画面で、以下の2行を実行するだけです。

```
/plugin marketplace add saladdays/agent-skills
/plugin install roundtable@saladdays-skills
```

インストールできたら `/roundtable:start [議題]` で円卓会議が始まります。

> これは Claude Code の「プラグイン」という仕組みです。1回入れたら、あとはずっと使えます。

### Cursor

Cursor のプラグインマーケットプレイスから `roundtable` を検索してインストール。

> [Cursor Marketplace](https://cursor.com/marketplace) に公開後はワンクリックでインストールできます。

### 他のAIツール

[SKILL.md](./skills/start/SKILL.md) をAIツールのスキルディレクトリにコピーしてください。

## 使い方

議題を伝えるだけです。URL、ファイル、テキスト、なんでも渡せます。

```
/roundtable:start このWebサイトを評価して: https://example.com
/roundtable:start 新規事業の戦略をレビューして（ドキュメントを添付）
/roundtable:start このコードのアーキテクチャを評価して
/roundtable:start チームの組織構造について議論したい
```

### 議論の流れ

1. **専門家パネルの提示** — 議題に応じた専門家チーム（3〜7名 + 批判者）が自動で選定されます。追加や変更も可能です
2. **独立分析** — 各専門家が他の意見を見ずに、独立して分析を行います
3. **構造化議論** — 意見が分かれた点について、根拠を示しながら議論が深まります
4. **レポート** — 合意点・対立点・少数意見・推奨アクションが構造化されたレポートで届きます

---

## レポートに何が書いてあるの？

| セクション | ひとことで言うと |
|---|---|
| **エグゼクティブサマリ** | 結論を3〜5行で。いちばん大事な発見と推奨アクション |
| **専門家パネル** | 誰がどんな視点で参加したか。★評価つき |
| **合意事項** | 全専門家が一致した点 |
| **主要な議論と結論** | 意見が分かれた点の結論と、その判断理由 |
| **少数意見** | 1人だけが指摘したが見逃してはいけない点 |
| **推奨アクション** | 優先順位つきのアクションリスト |
| **議論の限界** | この議論でカバーできなかった観点 |

ポイントは、**少数意見が構造的に保護されている**こと。多数派の結論と矛盾する意見も、根拠があれば必ずレポートに含まれます。「みんなが賛成しているから大丈夫」という思考停止を防ぎます。

---

## 議論の深度

議題の影響度やリスクに応じて、自動で深度が決まります（手動で指定も可能）。

| 深度 | Phase 2（議論） | セカンドチャンス | 用途 |
|---|---|---|---|
| Quick | スキップ | なし | 個人プロジェクト、軽い相談 |
| Standard | 最大2ラウンド | なし | 一般的な評価・レビュー |
| Deep | 最大2ラウンド | あり | 事業判断、高リスクな意思決定 |

専門家の人数は議題の複雑さに応じて3〜7名 + Devil's Advocate（批判者）。深度とは独立しています。

## なぜこう動くのか

### 専門家の選び方

「その分野に詳しい人」を集めるだけでは、視点が偏ります。Roundtable は5つの原則で専門家を選定します。

1. **失敗の検知者** — この議題が失敗したとき、最初に異変に気づくのは誰か。その視点を持つ専門家を入れる
2. **受け手の視点** — 作り手だけでなく、ユーザー・読者・利用者など影響を受ける側の視点を含める
3. **隣接分野の知見** — 見落とされがちな観点を持つ、隣の領域の専門家を加える
4. **意外な専門性** — 通常は招集されないが重要な知見を持つ分野がないか。組織設計に生態学者、AI開発に倫理哲学者——領域を横断する視点が本質的な発見をもたらすことがある
5. **構造的な批判者** — Devil's Advocate は全員が見過ごしている弱点を探す役割で、専門家とは別枠で常に配置する

また、同じ立場の専門家を重複させない設計になっています。たとえば「編集者とストラテジスト」はどちらも作り手側の視点。片方を読者側の専門家に替えることで、議論のバランスが変わります。

コードレビューだけでなく、事業戦略、デザイン評価、組織設計、記事の構成検討——どんなテーマでも、この原則に基づいて適切な専門家が自動で選定されます。

### なぜ議論を構造化するのか

AIに「複数の視点で議論して」と依頼すると、似たような意見が並んで穏やかに合意する結果になりがちです。これは人間の組織でも起きる現象で、心理学では Groupthink（集団浅慮）と呼ばれています。全員が「まあいいんじゃないか」と感じている間に、重大なリスクが見過ごされる。

Roundtable はこの問題に対し、議論のプロセス自体に構造を組み込むことで対処しています。

- **独立分析を先行させる** — 全員が他の意見を見ない状態で分析を完了させます。最初に聞いた意見に引きずられるアンカリング効果を排除するためです（デルファイ法の原則）
- **対立点に集中する** — 合意済みの論点を繰り返し確認するのではなく、意見が分かれている箇所に議論のリソースを集中させます
- **反復は2回まで** — マルチエージェント議論の研究で、3ラウンド以上の反復は少数派が多数派に迎合する Degeneration of Thought を招くことが確認されています。回数制限は意見の多様性を構造的に守る仕組みです
- **少数意見の構造的保護** — 多数派と矛盾する指摘であっても、根拠がある限りレポートに残します。「全員が賛成しているから正しい」という判断を防ぐためです

### 判断プロセスの透明性

レポートには結論だけでなく、そこに至る判断プロセスが記載されます。賛成側と反対側それぞれの根拠、どちらを採用しなぜそう判断したか、棄却した少数意見とその理由。判断の過程が見えることで、レポートを受け取った側が「この結論を信頼してよいか」を自分で評価できます。

これらの設計は、マルチエージェント議論（Multi-Agent Debate）の学術研究、デルファイ法、Kahneman の Adversarial Collaboration、Janis（1972）の Groupthink 研究に基づいています。詳細は [SKILL.md](./skills/start/SKILL.md) を参照してください。
