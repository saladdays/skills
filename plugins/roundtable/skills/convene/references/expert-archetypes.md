# 専門家アーキタイプ集

> 議題に応じて最適な専門家パネルを構成するための参照資料。
> キーワードリストではなく**原理**に基づいて動的に判断する。

## 専門家選定の原理

### 原理1: 失敗検知の原理

**「この議題で失敗したとき、誰が最初に気づくか？」**

その人の視点を持つ専門家がパネルに必要。例:
- Webサイトの表示崩れ → フロントエンドエンジニア
- ユーザー離脱率の悪化 → UXリサーチャー
- 法的リスク → 法務専門家

### 原理2: 盲点発見の原理

**「この議題で見落とされがちな観点は何か？」**

直接関連しないが、重要な視点を持つ専門家を1名含める。例:
- ECサイト評価 → アクセシビリティ専門家（忘れられがちだが法的義務になりつつある）
- 技術アーキテクチャ → ビジネスアナリスト（技術的に正しくても事業に合わない場合）
- 事業戦略 → エンドユーザー代弁者（経営視点に偏りがちな議論にユーザー視点を入れる）

### 原理3: 対立構造の原理

**Devil's Advocate は必ず1名含める。**

批判者の役割は「壊すこと」ではなく「盲点を発見すること」。
カトリック教会がこの役職を廃止した後、列聖の質が低下した歴史的事実がある。

### 原理4: 隣接知性の原理

**「一見関係なさそうだが、この議題に洞察を与えうる分野は何か？」**

- AI開発に哲学者を入れる（倫理・認識論の視点）
- プロダクト設計に行動経済学者を入れる（意思決定バイアスの視点）
- 組織設計に生態学者を入れる（複雑系・レジリエンスの視点）

この原理は Deep 深度でのみ適用する。Quick / Standard では直接関連する専門家に絞る。

## ドメイン別アーキタイプ

以下は**一例**であり、そのままコピーするものではない。
原理に基づいて議題ごとに動的に判断する。

### Web サイト・UI

| 専門家 | 主な観点 | 見逃しやすい盲点の例 |
|---|---|---|
| フロントエンドエンジニア | 実装可能性、パフォーマンス、レスポンシブ | 技術的に可能だがUXが悪いパターン |
| UI/UXデザイナー | ユーザー体験、情報設計、インタラクション | 美しいが使いにくいデザイン |
| アクセシビリティスペシャリスト | WCAG準拠、スクリーンリーダー対応、色覚多様性 | 健常者の視点だけで設計されたUI |
| コンテンツストラテジスト | メッセージング、トーン、情報の階層構造 | UIは良いがコンテンツが伝わらない |
| SEO/マーケティング | 検索性、コンバージョン、ユーザー獲得 | 美しいが見つけてもらえないサイト |

### モバイルアプリ

## Devil's Advocate の焦点の決め方

Devil's Advocate が批判する対象は、議題の性質に応じて設定する：

| 議題の性質 | Devil's Advocate の焦点 |
|---|---|
| 新しい提案 | 「なぜこれは失敗するか」 |
| 既存のシステム | 「なぜこれを変えるべきか」 |
| 戦略的判断 | 「この前提が間違っていたらどうなるか」 |
| デザイン | 「最も厳しいユーザーの視点で見たら何が問題か」 |
| コード | 「6ヶ月後に最も後悔する設計判断はどれか」 |

## 専門家数のガイドライン

| 深度 | 直接関連 | 隣接ドメイン | Devil's Advocate | 合計 |
|---|---|---|---|---|
| Quick | 2名 | 0名 | 1名 | 3名 |
| Standard | 2-3名 | 0-1名 | 1名 | 4名 |
| Deep | 2-3名 | 1名 | 1名 | 5名 |

5名を超えるとトークンコストに見合わない。
足りないと感じたら、専門家を増やすのではなく、既存の専門家の観点を広げる。
