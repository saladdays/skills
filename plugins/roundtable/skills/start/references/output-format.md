# 出力フォーマット

> 円卓会議レポートの出力テンプレートと記述ルール。

## 最優先ルール

**ソースに基づかない内容を生成しない。**
エージェントの分析結果に基づかない「まとめ」や「補足」を追加しない。
セクションを埋めるために情報を作り上げるくらいなら、そのセクションを省略する。

---

## 必須セクション

以下のセクションは深度に関わらず必ず含める。

### 1. エグゼクティブサマリ

3-5行で結論を述べる。忙しい読者がここだけ読めば全体像を掴めること。

```markdown
## エグゼクティブサマリ

[最も重要な発見を1行で]
[全体的な評価を1行で]
[最優先の推奨アクションを1行で]
```

### 2. 専門家パネル

参加した専門家の一覧。

```markdown
## 専門家パネル

| 専門家 | 役割 | 総合評価 |
|---|---|---|
| [名前/ロール] | [専門分野] | ★★★★☆ |
| [名前/ロール] | [専門分野] | ★★★☆☆ |
| Devil's Advocate | [批判の焦点] | — |
```

Devil's Advocate は「総合評価」を出さない（批判が役割なので評価は不適切）。

### 3. 合意事項

全専門家が一致した点。対立なく全員が同じ結論に達した事項をまとめる。

```markdown
## 合意事項

- [合意点1]: [根拠の要約]
- [合意点2]: [根拠の要約]
```

合意事項がない場合は「全員が一致した点はありませんでした」と明記する。

### 4. 主要な議論と結論

対立が生じた論点ごとに、議論の経緯と結論を記載する。

```markdown
## 主要な議論と結論

### [論点1のタイトル]

- **結論**: [採用された見解を1-2文で]
- **賛成側の根拠**: [具体的に。どの専門家の分析か明記]
- **反対側の根拠**: [具体的に。どの専門家の分析か明記]
- **判断理由**: [なぜこちらを採用したか。重み付けの根拠]
```

Quick 深度で Phase 2 をスキップした場合は「議論」ではなく
「異なる見解」としてまとめる。

### 5. 少数意見

1名だけが指摘したが重要な点。**このセクションは省略不可。**

```markdown
## 少数意見（見逃してはいけない指摘）

### [指摘内容]
- **指摘者**: [専門家ロール]
- **内容**: [具体的な指摘]
- **根拠**: [専門家が示した根拠]
- **対応**: [採用/棄却] — [理由]
```

少数意見がない場合は「少数意見はありませんでした」と明記する。

### 6. 推奨アクション

優先順位付きの行動提案。

```markdown
## 推奨アクション

1. **[最優先]** [アクション] — [根拠となった専門家の分析]
2. **[高]** [アクション] — [根拠]
3. **[中]** [アクション] — [根拠]
```

### 7. 議論の限界

この議論でカバーできなかった観点、追加調査が必要な点。
ユーザーが次に何をすべきかの示唆。

```markdown
## 議論の限界

- [カバーできなかった観点1] — [なぜカバーできなかったか]
- [追加調査が必要な点] — [具体的に何を調べるべきか]
```

## 条件付きセクション

以下は該当する場合のみ含める。

### Devil's Advocate の指摘（Deep のみ）

Devil's Advocate が特に鋭い指摘をした場合、独立セクションとして記載する。

```markdown
## Devil's Advocate の視点

[批判者が指摘した構造的な弱点・リスクの要約]
```

### 議論プロセス（Standard / Deep）

議論の透明性を担保するためのメタ情報。

```markdown
## 議論プロセス

- **深度**: [Quick/Standard/Deep] — [判定理由]
- **実行方式**: [サブエージェント並列 / Agent Teams 直接議論 / 逐次実行(fallback)]
- **フォールバック**: [該当する場合] 逐次実行に切り替えた理由と、ユーザーへの通知結果
- **Phase 2 参加者の選定理由**: [各対立点について、なぜその専門家を選んだか]
- **収束判定**: [どの条件で収束と判定したか]
- **設計上の限界**: サブエージェント方式での議論はデルファイ法に近い構造化レビューであり、
  エージェント間の動的な対話（MAD研究が前提とするステートフルな議論）とは性質が異なる
```

### セカンドチャンスの結果（Deep のみ）

Phase 3 のセカンドチャンスの結果。追加指摘がなかった場合も省略せず記載する。

```markdown
## セカンドチャンス（最終確認）

[追加指摘がある場合]
- [追加指摘]: [内容と対応]

[追加指摘がない場合]
追加指摘はありませんでした（N名が事前コミットと統合結果の一致を確認）。
```

---

## 記述ルール

### 透明性

- 各結論に「どの専門家の分析に基づくか」を明記する
- 対立の解決理由を明示する（「多数決で決めた」は不可。根拠の質で判断する）
- 議論の限界を正直に述べる

### 具体性

- 抽象的な評価（「概ね良好」）ではなく、具体的な指摘を含める
- 推奨アクションは実行可能なレベルまで具体化する
- 「〜すべき」だけでなく「なぜ」を併記する

### 不確実性の表現

確信度に応じて文体を使い分ける：

| 確信度 | 文体 | 使う場面 |
|---|---|---|
| 高 | 「〜である」「〜が必要」 | 全員合意 + 具体的根拠あり |
| 中 | 「〜と考えられる」「〜が推奨される」 | 多数派合意 or 条件付き結論 |
| 低 | 「〜の可能性がある」「〜を検討すべき」 | 少数意見 or 根拠が限定的 |
