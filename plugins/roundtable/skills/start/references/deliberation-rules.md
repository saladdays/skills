# 議論ルール

> 構造化された議論を運営するためのルール集。
> 学術的知見（MAD研究、デルファイ法、Groupthink回避）に基づく。

## 同意しすぎ問題（Sycophancy）への対策

LLMエージェントは人間と同様に「同意しすぎる」傾向がある。
MAD研究（2023-2025）で繰り返し確認されている。

### 効果がある対策

| 対策 | 仕組み |
|---|---|
| **対立ロールの明示的割り当て** | 「あなたは反対意見を述べる役割」と構造に組み込む |
| **独立意見生成フェーズ** | 他のエージェントの意見を見る前に自分の意見を出させる |
| **多数派情報を渡さない** | 「3人中2人が賛成」のような情報は少数意見を抑制する |
| **要約差分のみ伝える** | フル議事録ではなく、対立点の要約だけを渡す |

### 効果がない対策（使わないこと）

| 対策 | なぜ効果がないか |
|---|---|
| 「正しさを最優先してください」 | 研究（2025）で効果なし〜逆効果と確認済み |
| 自己申告の確信度による重み付け | LLMは議論開始時に72.9%、議論後に83.3%の過剰な確信を持つ |
| 3ラウンド以上の議論 | Degeneration of Thought: 初期エラーが固定化する |
| 同一プロンプトの複製 | Self-Consistency（多数決）以上の効果なし |

## 少数意見の保護メカニズム

### なぜ保護が必要か

MAD研究の重要な発見: **多数派の圧力は独立した修正を抑制する**。
つまり「3人中2人が賛成」という情報を渡すと、残り1人が正しい意見を変えてしまう。

歴史的事例: カトリック教会が Devil's Advocate を廃止した後、列聖審査の質が低下した。

### 保護ルール

1. **投票結果を見せずに反論を求める**: 「別の見解がある」とだけ伝え、誰が多数派かは伝えない
2. **少数意見を明示的にリストアップ**: Phase 3 で少数意見チェックは省略不可
3. **「棄却する理由」の記述を強制**: 少数意見を無視する場合、具体的な理由を書かなければならない
4. **疑わしきは採用**: 棄却理由が不十分なら、少数意見をレポートに含める

### 少数意見チェックの手順

> **注**: 少数意見チェックの判定基準は SKILL.md Phase 3-3 の 2x2 マトリクス
> （根拠有無×影響大小）が正である。以下の手順はその詳細な実施方法を示す。

```
1. 全エージェントの意見から、多数派と異なる意見を抽出する
2. 各少数意見について:
   a. 具体的な根拠があるか？ → ないなら棄却可（理由: 根拠なし）
   b. 他のエージェントがこの点に直接反論しているか？
      → 反論が具体的なら棄却可（理由: [反論の要約]）
      → 反論が抽象的なら採用する
   c. この少数意見が正しかった場合の影響は？
      → 影響が大きいなら採用する（たとえ根拠が弱くても）
3. 採用された少数意見は「少数意見セクション」に記載
4. 棄却された少数意見は棄却理由とともに記録（Deep のみ）
```

## 収束条件の判定

### 収束の定義

「これ以上議論しても新しい知見が出ない」状態。

### 判定方法

以下のいずれかを満たせば収束と判定する：

1. **全員合意**: 全エージェントが同じ結論に達した
2. **新規論点なし**: Round N の出力に、Round N-1 にない論点が含まれていない
3. **ハードキャップ**: Round 2 に到達

### Quick 深度の特別ルール

Quick では Phase 2（議論）をスキップ。独立分析の結果を直接統合する。
対立があっても議論せず、オーケストレーターが重み付けで解決する。

## 重み付けの方法

自己申告の確信度は使わない。代わりに以下の基準で重み付ける。

### 基準1: 根拠の具体性

| レベル | 例 | 重み |
|---|---|---|
| 高 | コード例、データ、仕様書の引用を含む | 最も重視 |
| 中 | 具体的な事例や経験に基づく | 標準 |
| 低 | 「一般的に」「ベストプラクティスとして」等の抽象的主張 | 低い |

### 基準2: 専門領域との一致度

その論点に最も関連する専門家の意見を重視する。
例: セキュリティの論点 → セキュリティ専門家の意見が最も重い。

### 基準3: 反論への応答品質

反論に対して具体的に応答できた意見を重視する。
反論を無視した、または「それでも私の意見は正しい」のみで応答した意見は軽視する。

## オーケストレーターの行動規範

オーケストレーターの役割は Phase によって異なる。

### Phase 1-2: プロセス管理者

議論の内容には関与せず、プロセスの制御に専念する。

**やるべきこと:**
- 議題と評価基準を明確にする
- 各エージェントに適切な情報を渡す
- 対立点を正確に要約する（下記の要約品質ルールに従う）
- 少数意見を保護する
- Devil's Advocate の批判品質を管理する（下記のDA品質管理ルール参照）

**やってはいけないこと:**
- 自分の意見を議論に入れる
- 多数派情報をエージェントに渡す
- DA に合意傾向や多数派の方向性を伝える（DA の独立した批判を汚染するため）
- エージェントに「正しい答えを出して」と指示する
- 3ラウンド以上の議論を行う

### Phase 3: 統合者

事前定義された基準（根拠の具体性・専門領域一致度・反論応答品質）に基づいて
判断を下し、最終的な推薦を出す。

**やるべきこと:**
- 重み付け基準を機械的に適用する
- 少数意見チェックを省略しない
- 判断の理由を明示する（「なぜこちらを採用したか」を必ず記述）

**やってはいけないこと:**
- 基準にない独自の判断軸を持ち込む
- 少数意見チェックを省略する

## 対立点の要約品質ルール

Phase 2 でオーケストレーターが対立点を要約する際、以下を守る。
要約がこのスキル全体の品質のシングルポイントオブフェイラーであることを自覚する。

### 要約に含めるべき情報の最低基準

- 元の主張の**結論**と**根拠**を両方含める（結論だけの要約は不可）
- 具体的なデータ・事例の引用は省略しない
- 要約によって一方の主張が有利/不利に見えるフレーミングをしない

### Deep 深度での追加ルール

- 要約とともに、元の出力の該当箇所を引用として渡す（要約のみへの依存を緩和）

## Devil's Advocate の品質管理ルール

Devil's Advocate の批判品質はオーケストレーターが管理する。
ペルソナの自己規律だけに依存しない。

### 過剰批判の判定基準

以下のいずれかに該当する場合、批判が過剰または低品質と判断する：

- 具体的な根拠や失敗シナリオを伴わない抽象的批判が3つ以上
- 他の全専門家の全発言に対して反対している（批判の焦点が絞れていない）
- 同じ批判を異なる言い回しで繰り返している

### 介入方法

- Phase 2 で過剰批判を検知した場合: 「最も構造的な弱点1つに絞って、具体的な失敗シナリオとともに再分析してください」と再指示
- Phase 3 の重み付けで: 根拠のない批判は明示的に低い重みとする（他の専門家と同じ基準を適用）

### 批判が不十分な場合の判定基準（下限）

以下のいずれかに該当する場合、批判が形骸化（形式的で機能していない）と判断する：

- 他の専門家の発言の言い換えにとどまり、独自の批判的視点がない
- 「潜在的なリスク」「考慮すべき点がある」等の指摘止まりで、具体的な失敗シナリオがない
- Quick 深度で DA の批判が全専門家の合意点と矛盾しない（批判として機能していない兆候）

### 介入方法（批判不十分の場合）

「もしこの提案/対象が最悪の形で失敗するとしたら、その原因を1つの具体的なシナリオで
描写してください」と再指示する

## Phase 2 での対立点の伝え方

### 良い例

```
あなたの分析では「モバイルファーストのレイアウトが適切」としていますが、
別の専門家は「デスクトップユーザーが70%を占めるためデスクトップ優先が適切」と
分析しています。

この対立について、あなたの観点から改めて分析してください。
意見を変える場合も、維持する場合も、具体的な根拠を示してください。
```

### 悪い例

```
他の3人の専門家はデスクトップ優先に賛成しています。
あなたの意見を再考してください。
```

→ 多数派圧力で少数意見が消える。
