# 専門家アーキタイプ集

> 議題に応じて最適な専門家パネルを構成するための参照資料。
> キーワードリストではなく**原理**に基づいて動的に判断する。

## 専門家選定の原理

### 原理1: 失敗検知の原理

**「この議題で失敗したとき、誰が最初に気づくか？」**

その人の視点を持つ専門家がパネルに必要。例:
- Webサイトの表示崩れ → フロントエンドエンジニア
- ユーザー離脱率の悪化 → UXリサーチャー
- 法的リスク → 法務専門家

### 原理2: 受け手の視点の原理

**「この議題の受け手・影響を受ける人は誰か？」**

作り手だけでなく、受け手（ユーザー、読者、利用者）の視点を含める。例:
- 記事レビュー → ターゲット読者（エンジニア、デザイナー等）
- API設計 → API利用者
- 事業戦略 → エンドユーザー代弁者

### 原理3: 盲点発見の原理

**「この議題で見落とされがちな観点は何か？」**

直接関連しないが、重要な視点を持つ専門家を含める。例:
- ECサイト評価 → アクセシビリティ専門家（忘れられがちだが法的義務になりつつある）
- 技術アーキテクチャ → ビジネスアナリスト（技術的に正しくても事業に合わない場合）

### 原理4: 隠れた専門知識の原理

**「この議題で、一般的には招集されないが、実は重要な知見を持つ分野はあるか？」**

Step 1-3 で専門家を選んだ後、この問いを明示的に自問する。
LLMの学習データには「この分野にはこの専門性が有効だった」という知見が大量に含まれている。
問われなければ出てこないが、問えば引き出せる。

既知の例（これらに限らない）:
- AI開発 → 倫理哲学者（Constitutional AI はまさにこの成果）
- プロダクト設計 → 行動経済学者（意思決定バイアス、ナッジ理論）
- 組織設計 → 生態学者（複雑適応系、レジリエンス）
- UX設計 → 認知心理学者（認知負荷、注意の科学）
- コンテンツ戦略 → 言語学者（フレーミング効果、語用論）
- セキュリティ設計 → 犯罪学者（攻撃者の動機モデル）

**重要: 候補が浮かばなければ無理に入れない。**
この原理はロジックで「入れるべき」と判断した場合のみ適用する。
全ての議題に隠れた専門家がいるわけではない。

### 原理5: 対立構造の原理

**Devil's Advocate は必ず1名含める（専門家枠とは別枠）。**

批判者の役割は「壊すこと」ではなく「盲点を発見すること」。
カトリック教会がこの役職を廃止した後、列聖の質が低下した歴史的事実がある。
DA はドメイン専門家ではなく批判の専門家であり、専門家数には含めない。

## ドメイン別アーキタイプ

以下は**一例**であり、そのままコピーするものではない。
原理に基づいて議題ごとに動的に判断する。

### Web サイト・UI

| 専門家 | 主な観点 | 見逃しやすい盲点の例 |
|---|---|---|
| フロントエンドエンジニア | 実装可能性、パフォーマンス、レスポンシブ | 技術的に可能だがUXが悪いパターン |
| UI/UXデザイナー | ユーザー体験、情報設計、インタラクション | 美しいが使いにくいデザイン |
| アクセシビリティスペシャリスト | WCAG準拠、スクリーンリーダー対応、色覚多様性 | 健常者の視点だけで設計されたUI |
| コンテンツストラテジスト | メッセージング、トーン、情報の階層構造 | UIは良いがコンテンツが伝わらない |
| SEO/マーケティング | 検索性、コンバージョン、ユーザー獲得 | 美しいが見つけてもらえないサイト |

## Devil's Advocate の焦点の決め方

Devil's Advocate が批判する対象は、議題の性質に応じて設定する：

| 議題の性質 | Devil's Advocate の焦点 |
|---|---|
| 新しい提案 | 「なぜこれは失敗するか」 |
| 既存のシステム | 「なぜこれを変えるべきか」 |
| 戦略的判断 | 「この前提が間違っていたらどうなるか」 |
| デザイン | 「最も厳しいユーザーの視点で見たら何が問題か」 |
| コード | 「6ヶ月後に最も後悔する設計判断はどれか」 |

## 専門家数のガイドライン

**パネルサイズは深度ではなく、議題が必要とする専門領域の数で決める。**
DA は専門家の枠とは別枠で常に1名。

| 議題の性質 | 直接関連 | 受け手 | 隣接ドメイン | DA | 合計 |
|---|---|---|---|---|---|
| 単一ドメインの明確な問題 | 2名 | 0-1名 | 0名 | 1名 | 3-4名 |
| 複数ドメインが絡む問題 | 2-3名 | 1名 | 0-1名 | 1名 | 5-6名 |
| 多くのステークホルダーに影響 | 3-4名 | 1名 | 1-2名 | 1名 | 7-8名 |

- **最小**: 3名 + DA = 4名
- **最大**: 7名 + DA = 8名
- 8名を超えるとトークンコストに見合わない

### 選定の落とし穴（避けるべきパターン）

- **同じ側の専門家を2名選ぶ**: 編集者 + コンテンツストラテジスト（両方「作り手」）
  → 代わりに編集者 + ターゲット読者（作り手 + 受け手）
- **受け手の視点を入れ忘れる**: 全員が「作る側」「評価する側」の専門家
  → 必ず「使う側」「影響を受ける側」を1名含める
- **DAを専門家枠に数える**: DAは批判の専門家であり、ドメイン専門家ではない
  → 専門家 N名 + DA 1名 として数える
