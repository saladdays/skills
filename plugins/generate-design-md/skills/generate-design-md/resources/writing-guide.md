# DESIGN.md 記述ガイド — Before/After

> DESIGN.md を書く際に陥りやすいパターンと、望ましい書き方の対比集。
> 設計原則に基づき、AIが一貫したビジュアル判断を行えるDESIGN.mdを書くための参照資料。

## セクション優先度（読む順序）

1. **TL;DR** (Section 1) — 必読。ここが間違っていれば全体がズレる
2. **Creative North Star** (Section 2) — 必読。メタファーの質が全体を決める
3. **Color Strategy** (Section 3) — 重要。ビジュアルの8割は色で決まる
4. **Typography** (Section 5) — 重要。可読性に直結
5. **Spacing & Layout** (Section 6) — 重要。レイアウトの流儀
6. **残り** (Section 7-16) — 参考。必要に応じて参照

---

## 修正の優先順位

DESIGN.md を修正するとき、以下の順番で確認すると効率的です。

1. **TL;DR** — ここが「違う」なら全体の方向性がズレている
2. **Creative North Star** — メタファーが腹落ちしているか
3. **Color Palette** — 色は最もすぐに「違い」が分かる
4. **Typography Scale** — フォントとサイズ感
5. **それ以外** — 使いながら気になった部分を随時修正

完璧にしてから使うのではなく、使いながら育てるのが DESIGN.md の正しい運用です。

## 1. TL;DR

5行以内で全体像を伝える。読み手が30秒で判断基準を掴めることがゴール。

### ❌ BAD

```markdown
## TL;DR
このプロダクトはモダンでクリーンなデザインを目指しています。
ユーザーフレンドリーなUIで、アクセシビリティにも配慮しています。
色はブランドカラーを使い、タイポグラフィはシステムフォントです。
レスポンシブ対応で、さまざまなデバイスで快適に使えます。
```

**理由:** 抽象的すぎて判断基準にならない。「モダン」「クリーン」は何も定義していないのと同じ。

### ✅ GOOD

```markdown
## TL;DR
- 北極星: 「静かな信頼感」— 堅すぎず、かかりつけ医のような安心
- カラー: ニュートラル基調、アクションは Blue-600 の1色
- タイポ: Inter、4段階、16px基準
- スペーシング: 8px単位、余白は広めに
- 避けること: 影の多用、3色以上の同時使用、12px以下のテキスト
```

**ポイント:** 各軸の具体的な判断基準が明示されており、AIはこの5行だけで大まかな方向性を掴める。

## 2. Creative North Star

メタファーはコンパスであり、ビジュアルの直訳ではない。「This does NOT mean」で過度解釈を防ぐ。

### ❌ BAD

```markdown
### Creative North Star
テーマは「森」です。自然で落ち着いた雰囲気を目指します。
```

**理由:** 「森」がUIにどう翻訳されるか不明。AIが木のテクスチャや葉っぱのアイコンを使い始めるリスクがある。

### ✅ GOOD

```markdown
### Creative North Star

**Metaphor:** 「深い森の木漏れ日」— 視界を遮るほどの情報はないが、空間には奥行きがある。

**This means:**
- サーフェス: 微細な明度差で奥行きを表現。フラットにはしない
- カラー: ニュートラルの中にグリーン系のアクセント1色
- 密度: 余白多め。1ビューに詰め込まない

**This does NOT mean:**
- 木や葉のテクスチャ・イラストを使うことではない
- 緑一色のUIではない
- アニメーションで光の揺らぎを再現することではない
```

**ポイント:** メタファーがUIの具体的特性に翻訳され、かつ誤解しやすい解釈が明示的に否定されている。

## 3. Global Constraints（Don't の書き方）

否定形には必ず肯定形の代替をペアにする。「やるな」だけでは AIは代替手段を推測するしかない。

### ❌ BAD

```markdown
### Global Constraints
- グラデーションを使わない
- 影を使いすぎない
- 派手な色を使わない
- アニメーションは控えめに
```

**理由:** 代替手段がないため、AIは「では何を使うか」を毎回推測する。「使いすぎない」「控えめに」は閾値が曖昧。

### ✅ GOOD

```markdown
### Global Constraints

- グラデーションを使わない → 代わりにソリッドカラーで表現する
- Material Design の elevation 体系を使わない → 代わりにサーフェス色差とボーダーで階層を表現
- 純黒 (#000000) をテキストに使わない → `--color-on-surface` (#1A1C1B) を使用
- アクセシビリティ: WCAG 2.2 AA 準拠を最低基準とする（コントラスト比 4.5:1）
```

**ポイント:** 各禁止事項に具体的な代替手段がセットになっており、AIが迷わず実装できる。

## 4. Color Strategy（Palette + Rules）

色は「原則 + スケール + 対応表」の3層で記述する。対応表はテーブル形式、Do/Don't列付き。

### ❌ BAD

```markdown
### Color
メインカラーは青（#2563EB）です。エラーは赤、成功は緑にしてください。
背景は白系で、テキストは黒系です。
```

**理由:** トークン名がないためコード上の一貫性が保てない。「黒系」「白系」は曖昧。Do/Don'tがないため誤用が防げない。

### ✅ GOOD

```markdown
### Color Strategy

**原則:** ニュートラル基調。アクセントカラーは1色に絞り、意味のある場面だけに使う。

### Palette

| Intent          | Token              | Value   | Do                    | Don't                |
|-----------------|--------------------|---------|----------------------|----------------------|
| Primary action  | `--color-primary`  | #2563EB | CTA、主要ボタン       | 装飾、背景全体       |
| Surface default | `--color-surface`  | #FAFAFA | ページ背景、カード    | テキスト色           |
| On surface      | `--color-on-surface`| #1A1C1B | 本文テキスト         | 純黒(#000)は使わない |
| Danger          | `--color-danger`   | #DC2626 | 削除、エラー          | 警告（warning を使う）|

### Rules

#### One-Accent Rule
同一ビューでアクセントカラーは `--color-primary` の1色のみ。
**Exception:** ステータス表示（danger/warning/success）は意味色として併用可。
理由: ステータス色は装飾ではなく情報伝達のため。

⚠️ Avoid: 3色以上を同一ビューで装飾に使う → アクセントは1色に絞る
```

**ポイント:** テーブルで全色が一覧でき、各色の用途と誤用が明確。名前付きルールで使い方の原則が参照可能。

## 5. Typography

スケール全体をテーブルで定義し、Reasoning列で「なぜそのサイズか」を示す。

### ❌ BAD

```markdown
### Typography
- h1: 32px bold
- h2: 24px bold
- h3: 20px semibold
- body: 16px normal
- small: 12px
```

**理由:** HTMLタグに紐づいており、セマンティクス（意味上の使い分け）がない。「h1をカード内見出しに使うか？」の判断ができない。Reasoning がなく、スケールの設計意図が不明。

### ✅ GOOD

```markdown
### Typography

**原則:** 見出しは存在感を、本文は可読性を最優先する。

| Context          | Token          | Font  | Size    | Weight | Line Height | Reasoning              |
|------------------|----------------|-------|---------|--------|-------------|------------------------|
| Hero headline    | `display-lg`   | Inter | 3rem    | 700    | 1.2         | ページの顔。視線の起点  |
| Page title       | `heading-xl`   | Inter | 2rem    | 600    | 1.3         | セクションの区切り      |
| Section heading  | `heading-md`   | Inter | 1.5rem  | 600    | 1.4         | コンテンツの構造化      |
| Body text        | `body-md`      | Inter | 1rem    | 400    | 1.6         | 長文の可読性基準        |
| Caption          | `body-sm`      | Inter | 0.875rem| 400    | 1.5         | 補助情報、低優先度      |
| UI label         | `label-md`     | Inter | 0.75rem | 500    | 1.4         | ボタン、フォームラベル  |

⚠️ Avoid: 0.75rem 未満のテキスト → 可読性とアクセシビリティの限界
```

**ポイント:** Context列でどの場面で使うかが分かり、Reasoning列で設計意図が伝わる。トークン名はコードで参照可能。

## 6. Spacing & Layout

スペーシングはスケール + ユースケースのテーブルで定義する。レイアウト方針は原則で記述する。

### ❌ BAD

```markdown
### Spacing
余白は適切に取ってください。
- 小さい: 4px
- 中くらい: 8px
- 大きい: 16px
- とても大きい: 32px
```

**理由:** 「適切に」は判断基準にならない。「小さい」「中くらい」はどの場面で使うか不明。スケール全体が見えない。

### ✅ GOOD

```markdown
### Spacing & Layout

**原則:** 8px基準のリズム。要素間の余白はコンテンツの関係性を表現する。
近い要素は関連が強い、遠い要素は独立している。

| Token       | Value | Use case                           |
|-------------|-------|------------------------------------|
| `space-1`   | 4px   | アイコンとラベルの間               |
| `space-2`   | 8px   | 関連する要素のグループ内           |
| `space-3`   | 12px  | フォームフィールド間               |
| `space-4`   | 16px  | カード内パディング                 |
| `space-6`   | 24px  | セクション内の要素間               |
| `space-8`   | 32px  | セクション間                       |
| `space-12`  | 48px  | 主要セクション間                   |
| `space-16`  | 64px  | ページレベルの区切り               |

### Layout Philosophy

#### Reading Room Rule
コンテンツ領域は常に最大幅を確保する。
- コンテンツ最大幅: 720px（長文の可読性に最適）
- サイドバーが必要な場合: 固定幅 240-280px

**Exception:** ダッシュボードではサイドバーに固定幅を割り当てる。
理由: 複数パネルの同時閲覧が業務効率に直結するため。
```

**ポイント:** 各トークンに具体的なユースケースがあり、レイアウト方針も名前付きルール + 例外で記述されている。

## 7. Component Patterns

コンポーネント個別のスタイル指定ではなく、カテゴリレベルの判断基準（デシジョンツリー）を書く。

### ❌ BAD

```markdown
### Components
- ボタン: 角丸8px、パディング12px 24px、フォントサイズ14px、色は青
- カード: 角丸12px、影あり、パディング16px
- モーダル: 幅480px、オーバーレイは黒50%
- テーブル: ヘッダーは灰色背景、行間にボーダー
```

**理由:** コンポーネント個別のpx指定はルール数が膨張する。新しいコンポーネントが登場するたびに追記が必要になり、スケールしない。

### ✅ GOOD

```markdown
### Component Patterns

#### Selection Guide

**選択肢の提示:**
- 3個以下 → Radio / SegmentedControl
- 4-7個 → Select
- 8個以上 → Combobox（検索付き）
- 理由: 認知負荷の閾値

**フィードバック:**
- 即時・軽微 → Toast（自動消去）
- 確認が必要 → Dialog
- 文脈内通知 → Inline Alert
- 理由: 中断の重さとユーザーの次のアクション

#### Composition Rules
- Card 内に Card をネストしない（フラットに並べる）
- Modal 内で別の Modal を開かない（Drawer または画面遷移で対応）

#### Density Guide

| Context       | Size | Inner Padding       | Use case                |
|---------------|------|---------------------|-------------------------|
| Dashboard     | sm   | `space-2` `space-3` | 情報密度優先            |
| Standard form | md   | `space-3` `space-4` | 入力の快適性            |
| Landing page  | lg   | `space-4` `space-6` | 視認性・余白の演出      |
```

**ポイント:** 「どの場面でどのコンポーネントを選ぶか」のカテゴリ判断を書いている。新しいコンポーネントが出ても、このデシジョンツリーで適切な選択ができる。

## 8. 名前付きルールと例外

ルールに名前をつけて参照可能にする。例外には「なぜ」を付ける。

### ❌ BAD

```markdown
### ルール
- コンテンツは広めに取る
- ただしダッシュボードは狭くてもいい
- ボーダーは基本使わない
- でもテーブルは使う
```

**理由:** ルールに名前がなく参照できない。例外の理由が書かれていないため、AIが他のケースに例外を拡大解釈するリスクがある。

### ✅ GOOD

```markdown
#### Content-First Rule
コンテンツ領域は常に最大幅を確保する。
**Exception:** ダッシュボードではサイドバーに固定幅を割り当てる。
理由: 複数パネルの同時閲覧が業務効率に直結するため。

#### Surface Layering Rule
背景色の明度差でUI階層を表現する。ボーダーではなく色の遷移が境界になる。
**Exception:** データテーブルの行間にはボーダーを使う。
理由: データの走査性は色差よりボーダーで向上するため。
```

**ポイント:** ルール名で会話中に「Content-First Rule に従って」と参照できる。例外に理由があるため、AIは「この例外は他のケースには適用しない」と判断できる。

## 9. 名前付きルールの独自性

ルール名はそのプロダクト固有の文脈から導出する。他プロダクトで見たルール名をそのまま流用しない。

### ❌ BAD

```markdown
#### Content-First Rule
コンテンツ領域は常に最大幅を確保する。

#### Surface Layering Rule
背景色の明度差でUI階層を表現する。
```

**理由:** 汎用的なルール名で、このプロダクトの文脈から導出されていない。AIが他のDESIGN.mdからコピーしたように見える。

### ✅ GOOD

```markdown
#### Cockpit Overview Rule
PMがダッシュボードの左上→右下で全プロジェクトの健全性を3秒以内に判断できること。

#### Ink-on-Paper Rule
階層の表現はボーダーで行う。影は「紙が浮いている」不自然さを生むため使わない。
```

**ポイント:** ルール名がプロダクトのメタファーから自然に導出されており、判断基準が固有の文脈に根ざしている。

## 10. ライブラリ依存の回避

特定のUIライブラリやCSSフレームワークに依存した記述を避ける。

### ❌ BAD

```markdown
### Components
- MUI の Button variant="contained" を使う
- Shadcn の Card コンポーネントを使う
- Tailwind の rounded-lg クラスを適用する
```

**理由:** 特定のライブラリに依存しており、技術スタックが変わると無効になる。ライブラリの選択は DESIGN.md の責務ではない。

### ✅ GOOD

```markdown
### Component Patterns
#### Selection Guide
選択肢が3個以下 → Radio / SegmentedControl
選択肢が4-7個 → Select
選択肢が8個以上 → Combobox（検索付き）
```

**ポイント:** コンポーネントの「種類」と「判断基準」を記述し、具体的なライブラリ実装には言及しない。

## 11. Line Height（行間の判断）

行間はプロダクトの性質と言語で変わる。「標準的」をコピーするのではなく、Reasoning で根拠を示す。

### ❌ BAD

```markdown
| Context | Token | Font | Size | Weight | Line Height | Reasoning |
|---|---|---|---|---|---|---|
| 本文 | `body-md` | Inter | 1rem | 400 | 1.5 | 標準的な行間 |
```

**理由:** 「標準的」は判断基準になっていない。何に対して標準なのか、なぜその値なのかが不明。

### ✅ GOOD（一例: 長文記事プラットフォーム）

```markdown
| Context | Token | Font | Size | Weight | Line Height | Reasoning |
|---|---|---|---|---|---|---|
| 記事本文 | `body-lg` | Noto Serif JP | 1rem | 400 | 1.8 | 日本語長文で1行35-45字を快適に読む行間 |
```

### ✅ GOOD（一例: ダッシュボード）

```markdown
| Context | Token | Font | Size | Weight | Line Height | Reasoning |
|---|---|---|---|---|---|---|
| UI本文 | `body-md` | Inter | 1rem | 400 | 1.5 | 短い説明文。情報密度を確保 |
```

**ポイント:** 値はプロダクトの性質と言語で変わる。「標準」をコピーするのではなく、Reasoning で根拠を示す。

## 12. 自由領域の記述

自由領域はリストで列挙するだけでは不十分。AIは自由領域でも保守的になりやすいため、トーンや方向性を添えることでクリエイティブな判断を促す。

### ❌ BAD

```markdown
### Creative Freedom
- 自由領域: ヒーローセクションの構成は自由
- 自由領域: ホバーエフェクトの演出は自由
```

**理由:** リストだけでは AI が保守的になり、無難なデフォルトに留まる。「自由」と言いつつ方向性がないため、結果的にテンプレート的な出力になる。

### ✅ GOOD（一例）

```markdown
### Creative Freedom
- 自由領域: ホバーエフェクトの演出。書斎で本を手に取る瞬間のように、静かだが確かな変化を
- 自由領域: ヒーローセクションの構成。最初の3秒で「ここは自分の居場所だ」と感じさせるトーンで
```

**ポイント:** トーンを添えることで AI に方向性を与える。自由領域でもメタファーに沿った判断ができるようになる。

## 13. 感情的インタラクションの State Handling

機能的操作と感情的操作を同じ処理にしない。ユーザーの感情が伴うインタラクションには、それに見合った演出を定義する。

### ❌ BAD

```markdown
| State | Visual Treatment | Reasoning |
|---|---|---|
| Hover | opacity 0.9 | 全要素共通のホバー |
```

**理由:** いいねボタンもフィルター変更ボタンも同じ opacity 0.9 で処理されている。機能的操作と感情的操作の区別がなく、ユーザーの感情的な満足が得られない。

### ✅ GOOD（一例）

```markdown
#### 汎用インタラクション
| State | Visual Treatment | Reasoning |
|---|---|---|
| Hover | 背景色を1段暗く | 操作可能の明示。控えめに |

#### 感情的インタラクション（スキ・フォロー等）
| State | Visual Treatment | Reasoning |
|---|---|---|
| Hover | アイコン色が `--color-primary` に変化 | 感情的アクションの予告 |
| Active (実行) | アイコン色変化 + scale バウンス (1.2→1.0) | 感情の「手応え」を返す |
```

**ポイント:** 「機能的操作と感情的操作を同じ処理にしない」が判断基準。具体的な演出（バウンス・色変化等）はメタファーに沿って決定する。

## 14. 書体戦略（Font Strategy）

書体の選択には必ず Reasoning を添える。1書体も Serif+Sans の組み合わせも正解になりうる。重要なのは「なぜその選択か」。

### ❌ BAD

```markdown
| Context | Token | Font | Size | Weight | Line Height | Reasoning |
|---|---|---|---|---|---|---|
| Hero headline | `display-lg` | Inter | 3rem | 700 | 1.2 | |
| Page title | `heading-xl` | Inter | 2rem | 600 | 1.3 | |
| Body text | `body-md` | Inter | 1rem | 400 | 1.6 | |
```

**理由:** 全行が Inter で Reasoning が空。なぜ1書体戦略なのか、なぜ Inter なのかが不明。テンプレートをコピーしたように見える。

### ✅ GOOD（一例: 記事プラットフォーム）

```markdown
| Context | Token | Font | Size | Weight | Line Height | Reasoning |
|---|---|---|---|---|---|---|
| 記事タイトル | `display-lg` | Noto Serif JP | 2rem | 700 | 1.4 | 作品の顔。文学的な第一印象 |
| UI本文 | `body-md` | Noto Sans JP | 1rem | 400 | 1.6 | 操作領域の機能性を確保 |

#### Serif/Sans Split Rule
コンテンツ領域は Serif、UI領域は Sans。読書と操作の境界を書体で暗示する。
```

### ✅ GOOD（一例: ダッシュボード）

```markdown
| Context | Token | Font | Size | Weight | Line Height | Reasoning |
|---|---|---|---|---|---|---|
| Page title | `heading-xl` | Inter | 1.5rem | 600 | 1.3 | Weight で階層を表現。装飾より視認性 |
| Body text | `body-md` | Inter | 0.875rem | 400 | 1.5 | 情報密度を確保。短文中心 |

#### Single Voice Rule
全UIを Inter で統一。Weight と Size のみで階層を表現し、書体の切り替えによる認知コストを排除する。
```

**ポイント:** 1書体も Serif+Sans も正解になりうる。Reasoning で「なぜその書体戦略か」を示すことが重要。

## 15. Hover/インタラクション表現

要素の役割ごとにホバーを判断する。全要素を同じ opacity で処理しない。

### ❌ BAD

```markdown
| State | Visual Treatment | Reasoning |
|---|---|---|
| Hover | 全要素 opacity 0.9 | ホバー状態 |
```

**理由:** カード・テキストリンク・アイコンボタン・CTA のすべてが同じ opacity 0.9。要素の役割に応じた適切なフィードバックが得られない。

### ✅ GOOD（一例）

```markdown
| Element | Hover Treatment | Reasoning |
|---|---|---|
| カード | 背景色を1段暗く + 微細な shadow 変化 | コンテンツ領域の境界を示す |
| テキストリンク | 下線出現 or 色変化 | テキスト内で操作可能を示す最小限の変化 |
| アイコンボタン | 背景の円形ハイライト | タップ領域の視覚化 |
| CTA ボタン | 背景色を hover 用トークンに切替 | 重要アクションの強調 |
```

**ポイント:** 要素の役割ごとにホバーを判断する。全要素を同じ opacity で処理しない。

## 16. 引用・特殊コンテンツブロック

引用スタイルはメタファーに依存する。「本文との区別が即座にできるか」が基準。

### ❌ BAD

```markdown
### Blockquote
- border-left: 4px solid gray
```

**理由:** 最小限の装飾のみで、メタファーとの接続がない。プロダクトの世界観から浮いたデフォルトスタイルになる。

### ✅ GOOD（一例: 記事プラットフォーム）

```markdown
#### Margin Note Rule
引用は「書斎の本棚から引き出した参照」として扱う。

| Block | Background | Border | Font | Reasoning |
|---|---|---|---|---|
| Blockquote | `--color-surface-inset` | left 3px `--color-border` | Serif italic | 本文と地続きだが、別の声であることを示す |
| Code block | `--color-surface-inset` | none, `radius-md` | Monospace | 技術的な引用。本文との書体差で区別 |
```

### ✅ GOOD（一例: ミニマリストプロダクト）

```markdown
#### Whisper Rule
引用は「声のトーンを落とす」表現にする。

| Block | Treatment | Reasoning |
|---|---|---|
| Blockquote | インデント 24px + フォントサイズ 0.9em | ボーダーや背景を使わず、余白とサイズだけで区別 |
| Code block | フォントサイズ 0.875em + monospace | 装飾を排除し、書体のみで区別 |
```

**ポイント:** メタファーに依存する。「本文との区別が即座にできるか」が基準。ボーダー+背景もインデントのみも、プロダクトの世界観に合っていれば正解。
