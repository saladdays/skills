# 専門視点タクソノミと自動選定

## 考え方

ベストプラクティスは「誰にとって」で変わる。同じクエリでも、セキュリティエンジニアと
UXデザイナーでは最適解が異なる。Perspective-aware IR (PIR) の知見に基づき、
検索クエリをトピック × 視点の組として扱う。

## 専門次元タクソノミ

**原理**: ベストプラクティスに影響する「利害・専門性・責任の軸」を特定する。
以下のタクソノミは **よく使うラベルの参考集** であり、網羅的ではない。
該当するものがなければ、トピックに応じて **仮ラベルを1〜2語で命名し、理由を明記** する。

### タクソノミ（非網羅的な参考集）

| 次元 | どのような視点か |
|------|-----------------|
| Security | 脅威モデル、攻撃ベクトル、認可・認証・データ保護 |
| Backend | API設計、DB設計、アーキテクチャ、スケーラビリティ |
| Frontend | UI実装、パフォーマンス、ブラウザ互換 |
| DevOps/SRE | CI/CD、監視、インフラ、信頼性 |
| Data | データモデリング、分析、パイプライン |
| QA/Testing | テスト戦略、品質保証 |
| UX/Design | 認知負荷、ユーザビリティ、情報設計 |
| PM/Product | 要件定義、優先順位、ロードマップ |
| Business | ビジネスモデル、ROI、市場分析 |
| Legal/Compliance | 規制、プライバシー、利用規約 |
| Education | 学習理論、教育方法、スキル開発 |
| Management | チーム運営、評価、組織設計 |
| Accessibility | WCAG、支援技術、インクルーシブ設計 |
| AI/DX | AIツール活用、プロンプト設計、開発者体験 |
| Medical/Health | 医学的エビデンス、公衆衛生、安全性 |
| Creator/Growth | クリエイター戦略、プラットフォーム最適化、オーディエンス構築 |
| Natural Science | 自然科学的原理、農学、環境科学 |
| Finance | 財務分析、投資理論、税務・会計 |

技術/非技術の区別は便宜的なもの。多くのトピックは両方にまたがる。
該当するラベルがなければ、トピック固有の専門軸を仮ラベルとして定義してよい。

## 自動選定メカニズム

### Step 1: 利害と影響の分析
キーワードマッチングだけでなく、以下を自問する：
- 「この推奨に従った場合、誰が影響を受けるか？」→ 影響を受ける立場 = 必要な視点
- 「この推奨が失敗した場合、どの領域に被害が出るか？」→ 被害領域 = 必要な視点
- 「この推奨を実行する人と、その結果を使う人は同じか？」→ 異なるなら両方の視点

### Step 2: よくある視点の組み合わせ（非網羅的な参考）
以下は頻出パターンの例。これ以外のトピックでも Step 1 の原理で視点を導出する。

| トピック例 | よく必要になる視点 |
|-----------|------------------|
| 認証・認可 | Security + Backend + （UIが絡めば UX） |
| CI/CD | DevOps + Backend + QA |
| API 設計 | Backend + UX（利用者体験）+ Security |
| チーム運営 | Management + PM |
| AIツール設計 | AI/DX + PM + （Security） |

### Step 3: スコアに基づく選定
- スコア上位 k 個を採用（k = 1〜3、深度レベルに応じて調整）
- Quick: 視点選定・クエリ変形は行わない（SKILL.md の Quick 定義に従う）
- Standard: k=2（主要2視点）
- Deep: k=3（3視点 + 必須ペアで追加される視点）

### Step 4: 冗長性の除去
スコアが近い視点が複数ある場合、クエリの意図に近い方を優先。
例: Backend と DevOps が両方高い → パイプライン寄りなら DevOps 優先

## 視点つきクエリの生成

各選定された視点について、検索クエリを変形する：

```
基本クエリ: "認証 ベストプラクティス 2026"

→ Security 視点: "認証 セキュリティ ベストプラクティス 脅威モデル OWASP 2026"
→ UX 視点: "認証 UX ベストプラクティス ユーザー体験 認知負荷 2026"
→ Backend 視点: "認証 実装 ベストプラクティス アーキテクチャ session JWT 2026"
```

## 視点統合のルール

### 単一視点の場合
そのまま通常のレポートとして出力。

### 複数視点の場合
1. 各視点のセクションを分けて記述
2. 視点間のトレードオフを明示
3. 矛盾がある場合は矛盾テンプレート（verification.md）を使用
4. 最終推奨はトレードオフ表で提示し、ユーザーの優先順位に委ねる

### トレードオフ表テンプレート
```markdown
| 観点 | 選択肢A | 選択肢B |
|------|---------|---------|
| Security | ✅ より安全 | ⚠️ 攻撃面が増える |
| UX | ⚠️ ステップが増える | ✅ よりシンプル |
| Backend | ✅ 実装しやすい | ⚠️ 複雑になる |
| 推奨状況 | セキュリティ重視の場合 | UX重視の場合 |
```

## 1視点で十分な判断基準

以下に該当する場合、複数視点は不要：
- ユーザーが明示的に役割を指定している（「SRE として」）
- 単一ドメインの定義済みタスク（「TypeScript の strict mode の設定」）
- 利害の衝突がない純粋な事実確認
- Quick 深度で十分と判断されたクエリ

## expert-review スキルとの連携

Deep 深度で専門視点が3以上必要な場合、`expert-review` スキルの
ペルソナチーム編成パターンを活用できる。expert-review は「レビュー」に、
本スキルは「調査」に特化している。
