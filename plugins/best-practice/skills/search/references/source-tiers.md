# ソースティアリングと権威評価

**原理**: ソースの信頼度は「プラットフォームの種類」だけでなく「誰が・何の目的で・
どの専門性に基づいて発信しているか」の総合で判断する。
以下のティアはデフォルトの出発点であり、「誰が言ったか」の評価で昇降格する。
リストにないソース種別でも、同じ原理（一次情報への近さ・検証可能性・利害関係）で判断する。

## ソースティア（デフォルトの序列）

### Tier 1: 最高信頼（公式・権威）
- 公式ドキュメント、API リファレンス
- RFC、W3C 仕様、ISO 標準
- OWASP、NIST、CERT 等のセキュリティ機関
- 政府・標準化団体の発行物
- 査読付き学術論文

**検証要件**: 日付・バージョンの照合のみ

### Tier 2: 高信頼（公式周辺・専門家）
- 公式 GitHub リポジトリ（Issue、PR、Release Notes）
- 公式ブログ・エンジニアリングブログ
- **認知された専門家** の個人ブログ（→「誰が言ったか」で昇格）
- カンファレンス講演（動画・スライド）
- プレスリリース・公式SNSアカウントの声明

**検証要件**: 日付照合 + 一次情報との整合確認

### Tier 3: 中信頼（コミュニティ・実績ベース）
- Stack Overflow の高投票回答（日付に注意）
- 高スター数のサンプルリポジトリ
- Zenn、Qiita、dev.to 等の技術メディア
- 業界メディア・専門誌
- パッケージのダウンロード数・スター数等の定量データ

**検証要件**: 日付照合 + 一次情報で裏取り + 2独立ソース

### Tier 4: 参考（個人・一般）
- 個人ブログ（専門性不明）
- 一般 GitHub リポジトリ
- Medium 等の一般プラットフォーム記事
- 書籍（出版年に注意）

**検証要件**: 必ず Tier 1-2 で裏取り

### Tier 5: 仮説生成用（SNS・非構造化）
- X（旧Twitter）個人アカウントの投稿
- Reddit、Hacker News のコメント
- Instagram、YouTube コメント
- 匿名掲示板

**検証要件**: 仮説・温度感の参考のみ。採用は禁止。必ず上位Tierで裏取り

## 「誰が言ったか」の評価（トピック条件付き権威）

プラットフォームの種別だけでなく、**発言者がそのトピックで実績があるか**を評価する。

### 昇格の条件（Tier を1段上げてよい場合）

1. **実体験に基づく発信（E-E-A-T の Experience）**
   - 同一製品・ツールの長期利用に基づく具体的なレビュー
   - 自ら実装・運用した経験に基づく知見
   - 例: 「1年間本番運用してわかった〜」→ Tier 3 でも証拠価値が上がる

2. **OSS メンテナが自分のプロジェクトについて語っている**
   - CODEOWNERS、Maintainer ロール、長期コミット履歴で確認
   - 例: React のコアチームが React について語る → Tier 1 相当

3. **そのトピックで複数のカンファレンス講演・論文がある人物**
   - 講演タイトル・論文キーワードがクエリトピックと一致
   - 例: Martin Fowler がリファクタリングについて → Tier 1 相当

4. **仕様策定に関与した人物**
   - RFC の Author、W3C Editor、TC39 メンバー等
   - 例: TC39 メンバーが JavaScript 仕様について → Tier 1 相当

5. **公式SNSアカウントの声明**
   - 企業・プロジェクトの公式アカウントによる告知・発表
   - 個人的意見ではなく組織としての声明であること
   - 例: @vercel の公式アナウンス → Tier 2 相当

### 降格の条件（Tier を信頼しすぎてはいけない場合）

1. **権威者がトピック外で語っている**
   - 有名エンジニアが専門外の経営論を語る → 一般人と同等
   - 芸能人・インフルエンサーが技術を語る → Tier 4 以下
   - **注意**: フォロワー数・知名度は権威の根拠にならない。
     その人がそのトピックで実績があるかが基準

2. **利害関係がある**
   - 自社製品を推奨している → 広告としてラベル
   - スポンサー付き記事 → 中立性を割り引く

3. **古い権威**
   - 過去の実績はあるが、そのトピックの最新動向から離れている
   - 例: 10年前の著書の内容を2026年にも同じく推奨 → 鮮度確認必須

## SNS ソースの特別扱い

### SNS が有用な場面
- 障害速報・互換性問題の早期発見
- ワークアラウンドの発見
- コミュニティの温度感・採用状況の把握
- 新しいツール・ライブラリの初期評価

### SNS が危険な場面
- セキュリティの推奨事項
- アーキテクチャの最終判断
- 事実の確定

### SNS 利用ルール
1. SNS の情報は「〜という声がある（未検証）」と明記する
2. 必ず上位 Tier で裏取りする
3. 投稿者がメンテナ・コミッタ等の場合は昇格ルール適用
4. スレッドの合意（反論・訂正の有無）を確認
5. 外部リンク（公式 Issue 等）がある場合はそちらを一次ソースとする

## メタデータの取得方法

Web検索結果から以下を可能な限り取得：
- `<meta name="author">` / JSON-LD `schema.org/Article` の `author`
- `datePublished` / `dateModified`
- ドメインのサブドメイン区分（`docs.` / `blog.` / `press.`）
- GitHub: Organization / Maintainer ロール / Release タグ
- SNS: 認証アカウント / プロフィールの役職

## ドメイン別の推奨ソース

| クエリタイプ | Tier 1 優先 | Tier 2 補完 |
|-------------|------------|------------|
| API・仕様・挙動 | 公式ドキュ、仕様書、RFC | 公式 GitHub Issue/PR |
| 実装ベストプラクティス | 公式ブログ、ガイド | 著名メンテナのブログ、カンファレンス講演 |
| セキュリティ | OWASP、CVE/NVD、ベンダーアドバイザリ | 公的 CSIRT |
| 文化・地域 | 現地語の一次報道・政府 | 学術論文 |
| 経営・ビジネス | 学術論文、HBR 等 | 実践レポート、ケーススタディ |
| 教育・学習 | 教育心理学の論文、教育機関、政府の教育計画 | 実践者のブログ（実績確認付き） |
| UX/デザイン | NNGroup等のUXリサーチ機関、デザインシステム公式 | 専門メディア（Smashing Magazine等） |
| マネジメント・組織 | Gallup等の大規模調査、組織心理学論文 | 実務書著者のブログ、カンファレンス講演 |
| SaaS/スタートアップ | 大手VC・決済基盤のリサーチ、業界ベンチマーク | SaaS専門メディア、実務家のケーススタディ |
| 医療・健康 | 査読付き医学論文、公的医療機関ガイドライン（WHO等） | 医療専門家のレビュー記事 |
| 法務 | 法令原文、弁護士会ガイドライン、判例 | 法律事務所のブログ（専門家確認付き） |
