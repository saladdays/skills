# Search Best Practice — ベストプラクティス調査

**「ベストプラクティス」を検索して、古い情報や個人の意見に振り回されたことはありませんか？**

技術選定、セキュリティ対策、価格設定——正しいやり方を知りたいのに、検索結果は玉石混交。3年前の記事が上位に来たり、ツールの宣伝がベストプラクティスとして紹介されていたり。

Search Best Practice はこの問題を解決します。

Web検索で最新のベストプラクティスを調査し、鮮度・信頼性・多角的視点を考慮した構造化レポートにまとめます。「原則（変わらないもの）」と「ツール・手法（すぐ変わるもの）」を分離して管理するので、来月には古くなる情報と、3年後も使える知見を区別できます。

---

## こんな方に

- **技術選定で迷っている方** — フレームワーク、ライブラリ、インフラ構成の「今の正解」を知りたい
- **新しいドメインに取り組む方** — 自分の専門外のベストプラクティスを効率よくキャッチアップしたい
- **チームの判断基準をつくりたい方** — 「なぜこのやり方がベストなのか」を根拠付きで説明したい
- **情報の鮮度が気になる方** — 古い記事に基づいて判断するリスクを避けたい

専門知識は必要ありません。知りたいことを伝えるだけで、調査から構造化までを自動で行います。

## 使い方

知りたいテーマを伝えるだけです。ドメインは自動で判定されます。

```
/search-best-practice:search Kubernetes シークレット管理のベストプラクティス
/search-best-practice:search SaaSの価格設定のベストプラクティス
/search-best-practice:search 子どもの読書習慣の身につけ方
```

### 調査の流れ

1. **自動判定** — テーマの鮮度要件、リスクレベル、必要な視点を自動で分析します
2. **Web調査** — 公式ドキュメント・学術論文・業界レポートを中心に、信頼性の高い情報源から調査します
3. **反証検索** — あえてアンチパターンや反対意見も探し、一面的な結論を防ぎます
4. **構造化レポート** — 原則とツールを分離した、すぐに使えるレポートが届きます

---

## レポートに何が書いてあるの？

| セクション | ひとことで言うと |
|---|---|
| **エグゼクティブサマリ** | 結論を簡潔に。いちばん大事なベストプラクティス |
| **原則（Layer S）** | 3年後も変わらない普遍的な原則。なぜそうすべきかの理由つき |
| **ツール・手法（Layer V）** | 今のベストな選択肢。鮮度と推奨度つき |
| **アンチパターン** | やってはいけないこと。よくある失敗とその理由 |
| **判断マトリクス** | 状況に応じた選択指針（「こういう場合はAを選ぶ」） |
| **情報源** | 参照した全ソースと信頼性評価 |

ポイントは、**原則とツールが分離されている**こと。「Kubernetes のシークレットは外部ストアに保存すべき」（原則 = 安定）と「Vault を使う」（ツール = 変わりうる）を区別できるので、ツールが変わっても判断基準は残ります。

## 対応ドメイン

技術・ビジネス・教育・医療・法務・デザイン・クリエイティブ等、あらゆるドメインに対応。原理ベースで設計されているので、キーワードリストに依存せず、未知のドメインにも適応します。

15 トピック × 3 ラウンドのテストで全件完走を確認済み。

## 調査の深度

テーマのリスクや鮮度要件に応じて、自動で深度が決まります。

| 深度 | 調査の深さ | 用途 |
|---|---|---|
| Quick | 主要ソース中心に簡潔に | 一般的な質問、低リスク |
| Standard | 複数ソース + 反証検索 | 技術選定、業務プロセス |
| Deep | 網羅的調査 + リスク分析 | セキュリティ、医療、法務、金融 |

## なぜこう動くのか

### 情報の「持ってきかた」

Web検索で上位に来る記事が正しいとは限りません。Search Best Practice は、情報を集める段階で3つのフィルターをかけています。

1. **誰が言っているか** — 公式ドキュメントや学術論文は、個人ブログより信頼できます。情報源を「公式 → 専門機関 → 実務者 → 個人」の順にランク付けし、信頼度の高いものを優先します
2. **いつの情報か** — セキュリティのベストプラクティスは半年で変わることがあります。テーマごとに「この情報は何ヶ月以内なら信頼できるか」を判定し、古すぎる情報には注意マークをつけます
3. **反対意見も探す** — 「Reactがベスト」と結論を出す前に、あえて「Reactの問題点」も検索します。片方の意見だけ集めると、都合のいい情報ばかりになってしまうからです

これは情報リテラシーの分野で SIFT（Stop, Investigate, Find, Trace）や CRAAP テスト（Currency, Relevance, Authority, Accuracy, Purpose）と呼ばれる検証手法を、AIの調査プロセスに組み込んだものです。

### 「原則」と「ツール」を分ける理由

ベストプラクティスには、すぐに古くなるものと、長く使えるものがあります。

たとえば「パスワードはハッシュ化して保存する」は原則です。10年前も10年後も変わりません。一方「bcrypt を使う」はツールの話で、将来もっと良い方法が出てくるかもしれません。

この2つを混ぜてしまうと、ツールが変わったときに原則ごと捨ててしまったり、古いツールに固執してしまったりします。分けておけば、ツールが変わっても「なぜそうすべきか」という判断基準は手元に残ります。

### この考え方自体が大事なスキル

Search Best Practice が出すレポートの内容はもちろん役に立ちますが、それ以上に大事なのは、レポートが示す **情報の評価の仕方そのもの** です。

「誰が言っているか確認する」「いつの情報か確認する」「反対意見も探す」「原則とツールを分けて考える」——これは、AIに頼らなくても使える考え方です。レポートを読むうちに、自分でベストプラクティスを調べるときの判断力も自然と身につきます。
